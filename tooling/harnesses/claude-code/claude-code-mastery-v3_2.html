01 Configure The foundation layer You own the config

CLAUDE.md, hooks, permissions, MEMORY.md. The infrastructure every team should commit before writing a single prompt — and keep updating forever as Claude makes new mistakes.

The Level 1 Test
"Do you have a CLAUDE.md in every active repo, reviewed as carefully as code, and hooks that enforce what CLAUDE.md can only suggest?"
Primitives
CLAUDE.md
Advisory. ~70-80% reliable. Behavioral guidance, not enforcement. Run /init to generate from your codebase.
Hooks
Deterministic. 100%. 25+ lifecycle events: PreToolUse, PostToolUse, Stop, SubagentStop, PreCompact, SessionStart.
Permissions
Deny-first evaluation. Denied files are invisible to Claude — not just blocked. allow / ask / deny.
MEMORY.md
Session state handoff. Claude reads at session start, writes at end. Survives across conversations.
Slash commands
.claude/commands/*.md — reusable prompt procedures. Team-shared, version-controlled like code.
@imports
@docs/testing.md in CLAUDE.md loads domain docs conditionally without bloating every session.
Patterns
P1
The living CLAUDE.md — correction loop
Encode every wrong assumption immediately. The file compounds over time like interest.
+

Boris Cherny's team rule: when Claude does something wrong, add it to CLAUDE.md so it doesn't repeat. Don't just fix the output and move on — fix the system. The six-month CLAUDE.md built this way is worth more than any prompt.

Root file stays under 100 lines of invariants. Use @imports for domain context that would bloat every session. Every line earns its place — ask "would removing this cause Claude to make a mistake?"

Use the WHAT/WHY/HOW framework for each rule:
• WHAT: "Never use --foo-bar flag"
• WHY: "Creates write locks in DB layer"
• HOW: "Use --baz instead (docs/db.md)"

Negative-only rules without alternatives get Claude stuck. Always provide the redirect.

Diagnostic prompt for persistent mistakes: "Stop. Before fixing the code — explain what rule you're supposed to follow here, why you keep getting it wrong, and propose exactly what to add to CLAUDE.md to prevent this permanently."

CLAUDE.md template (~100 lines)
# [Service Name]

## Identity (one line)
REST API. Node 20, TypeScript strict,
PostgreSQL/Drizzle. Deployed AWS ECS.

## Commands — use these exactly
- Test: `pnpm test`
- Test single: `pnpm test [path]`
- Lint/fix: `pnpm lint:fix`
- Build: `pnpm build`
- Migrate (dry-run): `pnpm db:migrate`

## Architecture rules
- Domain logic ONLY in src/domain/
- HTTP layer ONLY in src/api/
- DB access via repos in src/repos/ only
- Secrets from env — NEVER hardcode

## Code style
- Named exports only (no default exports)
- Throw typed errors from src/errors/
- async/await — no raw Promise chains

## Known wrong defaults
- ALWAYS pnpm, NEVER npm or yarn
- Import @company/core not @company/utils
  (deprecated since v3.2)
- logger from src/lib/logger.ts, not console
- NEVER log request bodies, tokens, or PII

## Compaction policy
When context compacts, preserve:
- Current task definition
- Files modified this session
- Errors encountered + resolutions
- Architectural decisions made

@docs/error-handling.md
P2
The four essential hooks — full production config
Auto-format, destructive block, file protection, test-gated stop. The deterministic enforcement layer.
+

The critical distinction: CLAUDE.md is advisory (~70-80% followed). Hooks are deterministic — they fire every time, cannot hallucinate, cannot be reasoned around.

Exit codes: exit 0 = allow, exit 2 = block (even in bypassPermissions mode), exit 1 = warn and continue. Security hooks must exit 2 or they provide no enforcement.

The stop_hook_active guard is mandatory on Stop hooks. Without it: Claude stops → hook exits 2 → Claude works → stops → hook fires again → infinite loop. Always check this field first and exit 0 if the hook already ran this stop.

Performance budget: hooks on PreToolUse/PostToolUse fire on every matched tool call. Keep each hook under 200ms or the session degrades. Profile with time before deploying.

.claude/settings.json
{
  "permissions": {
    "allow": ["Bash(pnpm test:*)","Bash(pnpm lint:*)",
      "Bash(git status)","Bash(git diff:*)",
      "Bash(git add:*)","Bash(git commit:*)",
      "Read","Glob","Grep"],
    "ask": ["Bash(git push:*)","Bash(pnpm db:migrate:*)"],
    "deny": ["Read(./.env)","Read(./.env.*)",
      "Read(./secrets/**)","Bash(curl:*)"]
  },
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit|MultiEdit",
      "hooks": [{"type":"command",
        "command":"npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
      }]
    }],
    "PreToolUse": [
      {"matcher":"Bash","hooks":[{"type":"command",
        "command":"\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/guard-bash.sh"
      }]},
      {"matcher":"Write|Edit","hooks":[{"type":"command",
        "command":"\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-files.sh"
      }]}
    ],
    "Stop": [{"hooks":[{"type":"command",
      "command":"INPUT=$(cat); [ \"$(echo $INPUT|jq -r '.stop_hook_active')\" = 'true' ] && exit 0; pnpm test --silent 2>&1 | tail -5 || exit 2"
    }]}],
    "SessionStart": [{"hooks":[{"type":"command",
      "command":"echo \"{\\\"additionalContext\\\":\\\"Branch: $(git branch --show-current). See .claude/sprint-context.md\\\"}\""
    }]}]
  }
}
P3
MEMORY.md session handoff protocol
Session state that survives across conversations. Claude reads it cold and knows exactly where you left off.
+

CLAUDE.md carries static project knowledge. MEMORY.md carries session-specific state — what you decided last session, what's in-progress, what's blocked. Over months it becomes a decision log more valuable than git history.

The most valuable MEMORY.md entries are the "we tried X but it caused Y" decisions. These are exactly the assumptions Claude re-makes every fresh session without this file. Negative decisions compound too.
.claude/commands/session-end.md
Before ending this session, update .claude/MEMORY.md:

## Last updated: [date/time]

## Completed this session
[Specific tasks with file paths]

## In-progress state
[What's partially done; files in transition]

## Decisions made
[Architectural choices, rejected approaches, and
WHY they were rejected — the critical part]

## Blockers
[What's blocked and on what]

## Next session start prompt
[Exact prompt to resume efficiently]

Update the existing file. 3-5 bullets max per section.
.claude/commands/session-start.md
Before anything else:
1. Read .claude/MEMORY.md if it exists
2. Summarize current state in 2 sentences
3. Ask: "Resume [in-progress work] or new task?"

Then wait for direction before taking any action.
P4
PreCompact context preservation + PostCompact re-injection
Back up working context before Claude's memory compresses. Re-inject after. Async — never blocks execution.
+

When Claude's context fills and auto-compacts, the working memory is compressed. PreCompact fires first — use it to write a backup before compression. PostCompact fires after — use it to re-inject context that compaction might have lost.

Also shape what survives compaction via the CLAUDE.md compaction policy section — Claude uses this to prioritize what to preserve in its compaction summary.

async: true (released Jan 2026) runs the hook in the background without blocking execution. Use async for backups, logging, and notifications. Never async for security gates — those must block synchronously.
.claude/settings.json (PreCompact + PostCompact)
{
  "hooks": {
    "PreCompact": [{
      "hooks": [{
        "type": "command",
        "async": true,
        "command": "cp ~/.claude/projects/$(basename $PWD)/$CLAUDE_SESSION_ID.jsonl ~/.claude/backups/session-$(date +%s).jsonl 2>/dev/null || true"
      }]
    }],
    "PostCompact": [{
      "hooks": [{
        "type": "command",
        "command": "echo '{\"additionalContext\":\"Context just compacted. Re-read .claude/MEMORY.md if you need task context that may have been lost.\"}'"
      }]
    }]
  }
}
P5
Wrong defaults to encode before session one
The assumptions Claude ships with that will bite you. Commit these to CLAUDE.md on day one.
+
CLAUDE.md — known wrong defaults section
## Known wrong defaults

### Package manager
ALWAYS pnpm. NEVER npm or yarn.

### Testing
Run only affected file: `pnpm test [file]`
Full suite only when explicitly asked.

### Database
NEVER run migrations against production.
`pnpm db:migrate` is dry-run by default (hook enforced).
Actual execution: `pnpm db:migrate --execute`

### Git
NEVER git push --force. Use --force-with-lease.
Commits: feat|fix|docs|refactor|test|chore(scope): desc

### File creation
NEVER create a service without checking src/services/ first.
Check for existing patterns before anything new.

### Error handling
NEVER throw plain Error(). Use src/errors/index.ts:
ValidationError, NotFoundError, AuthError, ExternalServiceError.

### Logging
NEVER console.log. ALWAYS logger from src/lib/logger.ts.
NEVER log bodies, tokens, or anything with user PII.
Anti-patterns at this level
The kitchen sink session
One task, then another, then another. Context fills with noise. Use /clear between unrelated tasks.
Correcting over and over
Fixing output without encoding the fix in CLAUDE.md. Same mistake recurs next session. Encode once.
CLAUDE.md over 200 lines
Claude attends ~150 instructions reliably. Past 200 lines, rules get lost in noise. Every line must earn its place.
Rules without alternatives
"Never use --foo-bar" gets Claude stuck. "Never use --foo-bar; prefer --baz (docs)" gives it a path.
02 Program Context architecture You own the workflow

Subagents, skills, agent teams, worktrees. The key insight: context isolation is the primary value — not just parallelism. A 40%-full focused agent outperforms a 90%-full general one on every reasoning task.

The Level 2 Test
"When exploring an unfamiliar module, do you delegate to a subagent — or let the exploration logs pollute your main session context?"
Primitives
Subagents
Isolated context windows. Only the summary returns — not exploration logs, failed attempts, or intermediate reasoning.
Skills
.claude/skills/name/SKILL.md — procedures that load on-demand. Body costs zero tokens until invoked.
Agent teams
Bidirectional — teammates message each other directly. Requires CC v2.1.32+. Enable via settings.json env.
Git worktrees
Each session in its own branch + directory. Zero context bleed between parallel work. 4-8 concurrent reliably.
/compact at 70%
Proactively compact before the model degrades, not reactively when mistakes start appearing.
Forks
Inherit full conversation history. Use when a subagent would need too much background context to start fresh.
Patterns
P1
Codebase explorer subagent — pure research isolation
All exploration in its own context window. Only a clean structured summary returns to your session.
+

Exploring an unfamiliar module generates noise: grep output, file reads, dead-ends, intermediate reasoning. Done in your main session, that noise fills context and degrades all subsequent work. The subagent takes all of it and returns only the distilled summary.

Use for: unfamiliar modules before touching them, third-party API research, dependency analysis, "how does X work" investigations. Never use for implementation — research only. The moment you start writing code in a research subagent, you've broken the isolation model.
.claude/agents/codebase-explorer.md
---
name: codebase-explorer
description: >
  Use when you need to understand how a module,
  pattern, or system works before implementing.
  Explores independently, returns structured
  summary. Do NOT use for implementation.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You investigate code and return clean summaries.
You do NOT write production code.

When invoked, receive a research question.
Explore: read files, trace call chains,
find tests as behavior documentation.

Return structured summary:
1. How it works (2-3 paragraphs)
2. Key files and their roles (with paths)
3. Patterns to follow when adding to this system
4. Things NOT to do (common mistakes)
5. Open questions the code didn't answer
P2
Writer/reviewer loop — eliminating author bias
One session writes. A fresh session reviews. Fresh context has no anchor to the code it just wrote.
+

Claude reviewing its own code always approves it. The author bias is structural — the same context that wrote the code will rationalize it during review. A fresh session with no writing history is genuinely adversarial.

For payment and auth code: use a dedicated code-reviewer agent (model: opus) as the reviewer. Security misses are expensive enough to justify the cost difference. Add to CLAUDE.md: "Before committing anything in src/payments/ or src/auth/, MUST invoke the code-reviewer agent."
.claude/agents/code-reviewer.md
---
name: code-reviewer
description: >
  Adversarial code review. Invoke after any
  payment, auth, or data mutation implementation.
  Proactively before any commit in
  src/payments/ or src/auth/.
tools: Read, Glob, Grep, Bash
model: opus
---

You find problems. Not validate that work looks good.

When invoked:
1. Run `git diff HEAD` — see what changed
2. Read every changed file completely
3. For each file, check:
   - Security: input validation, auth bypass, PII
   - Correctness: edge cases, concurrent access
   - Performance: N+1 queries, unbounded loops
   - Tests: do they actually test the behavior?

Output format — be specific or say nothing:
CRITICAL (block merge): [issue] at [file]:[line]
IMPORTANT (fix before ship): [issue] at [file]:[line]

If nothing critical: say so with your reasoning.
P3
Vertical slices, not horizontal phases
Force tracer bullets through the full stack. Catch integration problems on task one, not the last task.
+

Claude defaults to horizontal phases: all DB first, then all API, then all UI. You don't see working code until the final phase — and integration problems appear too late to fix cheaply.

Vertical slices cut through all layers for one complete user-visible feature. Each slice is independently runnable. Integration problems surface on slice one when they're still cheap.

Always include a review checkpoint between slices. "After Slice 1 passes all tests, stop and let me review before Slice 2." This creates a natural human-in-the-loop gate without interrupting flow within a slice.
Vertical slice prompt pattern
Implement [feature] as vertical slices.

Slice 1: "User can see unread notification count"
- DB: notifications table (id, user_id, type,
  read_at, created_at)
- Service: getUnreadCount(userId)
- API: GET /api/notifications/count
- Frontend: NotificationBadge using the endpoint
- Test: integration test for the full slice

Stop after Slice 1 passes all tests.
Wait for my review before proceeding.

Slice 2: "User can mark a notification as read"
[continue...]

Each slice must be independently demoed.
Do not proceed past a checkpoint without approval.
P4
Skill frontmatter — once, fork, hooks, auto-trigger
Four advanced fields most developers never use. Run once per session. Isolate in fork. Attach scoped hooks.
+

once: true — the skill body executes only once per session. Ideal for expensive setup: loading architecture docs, reading schema files. Subsequent invocations are no-ops.

context: fork — runs in a forked subagent context. Skill's verbose processing stays isolated; your main context only sees the result.

hooks in frontmatter — scoped to the skill's lifecycle only. A PreToolUse hook inside a skill fires only for that skill's tool calls, not the entire session.

Description precision is everything for auto-trigger. "Use when the user asks to review code, after writing new code, or mentions wanting a PR review" triggers reliably. "Code review stuff" never triggers. The description is how Claude decides whether to invoke the skill.
.claude/skills/load-architecture/SKILL.md
---
name: load-architecture
description: >
  Load and internalize project architecture docs.
  Use at the start of every session and whenever
  architecture context is needed.
once: true
---

Read and internalize these files completely:
- docs/architecture.md
- docs/domain-model.md
- docs/adr/ (all architectural decision records)

After reading, summarize in 3 bullets:
what this system does, key architectural
constraints, and primary data flows.
Then confirm you're ready to work.

name: secure-ops

P5
Git worktrees with role discipline
Each task in its own branch and directory. The failure mode is role blur — sessions drifting into each other's files.
+

Two Claude sessions on the same branch means the moment one edits a file, the other's context is stale. Worktrees give each session its own filesystem, branch, CLAUDE.md view, and conversation history.

Role discipline is the failure mode. The moment Session A starts touching files that belong to Session B, stop, commit what's safe, and reset. Parallel sessions fail when they blur — not when they start. Assign explicit file boundaries per session.

Advanced: use /plan in the main session to decompose a feature, then spawn one worktree per task. Each worktree runs independently, you review diffs as they complete, merge sequentially. Closest thing to "multiple engineers on one feature" AI tooling has produced.

Parallel worktree setup + agent teams
# Create isolated worktrees
git worktree add ../myrepo-feat-auth feat/oauth
git worktree add ../myrepo-fix-deploy fix/timeout
git worktree add ../myrepo-docs docs/api-update

# Open Claude in each (separate terminals)
cd ../myrepo-feat-auth && claude
# → "Focus ONLY on src/auth/ — no other dirs"

cd ../myrepo-fix-deploy && claude
# → "Focus ONLY on src/deploy/healthCheck.ts"

# Cleanup after merge
git worktree remove ../myrepo-feat-auth

03 Orchestrate Pipeline engineering You own the pipeline

Headless mode, GitHub Actions, input rewriting, grading loops. Your application now runs agents — no human watching, deterministic output, verifiable quality, budget caps in place.

The Level 3 Test
"Does your CI pipeline run Claude automatically on every PR with token budget caps, structured JSON output, and dedup to avoid comment spam across runs?"
Primitives
claude -p --bare
Non-interactive. --bare = no runner state bleed. Without it, CI behavior depends on invisible ~/.claude/ state per runner.
stream-json output
Every tool call, decision, error as a structured JSON line. Essential for CI observability and debugging.
Input modification
PreToolUse hooks silently rewrite tool inputs before execution. Claude uses the safe version, never knows.
Grading loops
Draft → grade (fresh context, no author bias) → retry. 10.1% quality lift with no model change.
--allowedTools
The only hard enforcement boundary. CLAUDE.md constraints are advisory; this whitelist is a technical limit.
--max-turns + USD cap
Hard loop caps and dollar ceilings. Without these, CI runs are unbounded. You will have a billing event.
Patterns
P1
The grading loop — quality without model change
Draft in session 1. Grade against a rubric in fresh context. The grader has no author bias. Retry only on failure.
+

Anthropic's Outcomes feature (Managed Agents, public beta) implements this pattern natively. You can implement it manually today. The key: a grader in a fresh context window is not influenced by the reasoning that produced the draft.

A well-written rubric is worth more than three additional subagents. The rubric forces you to articulate what "good" looks like — which is actually the hard problem. Anthropic's internal benchmarks show 10.1% improvement on pptx, 8.4% on docx, purely from adding the grading layer with no model change.
.claude/commands/graded-review.md
You will do a graded PR review.
Rubric is at: .claude/rubrics/pr-review.md

Pass 1 — Draft:
Review the current diff. Write your review
to .claude/review-draft.md.

Pass 2 — Grade (fresh read):
Read .claude/rubrics/pr-review.md carefully.
Read .claude/review-draft.md as if you
didn't write it. Score adversarially.
Write grade JSON to .claude/review-grade.json:
{
  "specificity": 1-5,
  "actionability": 1-5,
  "false_positive_rate": 1-5,
  "coverage": 1-5,
  "pass": boolean,
  "weakest_dimension": "...",
  "retry_instruction": "Focus on [gap]"
}

Pass 3 — Retry if failed:
If pass=false: rewrite targeting weakest_dimension.
Re-grade. Maximum 3 attempts total.
If still failing: output best version and flag:
REVIEW_NEEDS_HUMAN_ESCALATION
P2
Input modification — transparent sandboxing
Hooks that silently rewrite Claude's tool inputs before execution. Claude never knows the redirect happened.
+

Since CC v2.0.10, PreToolUse hooks can return modified JSON that Claude uses instead of its original input. Three production patterns: branch safety on push, dry-run on migrations, and stripping --no-verify from commits.

This is not configuration or enforcement — it's structural safety that can't be reasoned around because Claude doesn't know it's happening. The most powerful hooks are the ones that work invisibly.
.claude/hooks/safe-push.sh
#!/bin/bash
# Silent input modification — Claude never sees this
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# Redirect push to main → current branch
if echo "$COMMAND" | grep -qE 'git push.*(origin main|origin master)'; then
  BRANCH=$(git branch --show-current)
  SAFE=$(echo "$COMMAND" | sed "s/origin main/origin $BRANCH/g")
  echo "$INPUT" | jq --arg c "$SAFE" '.tool_input.command = $c'
  exit 0
fi

# Inject --dry-run on migrations unless --execute present
if echo "$COMMAND" | grep -q 'pnpm db:migrate' && \
   ! echo "$COMMAND" | grep -q '\-\-execute'; then
  echo "$INPUT" | jq --arg c "$COMMAND --dry-run" '.tool_input.command = $c'
  exit 0
fi

# Strip --no-verify (force hooks to always run)
if echo "$COMMAND" | grep -q 'git commit.*--no-verify'; then
  SAFE=$(echo "$COMMAND" | sed 's/--no-verify//g')
  echo "$INPUT" | jq --arg c "$SAFE" '.tool_input.command = $c'
  exit 0
fi

echo "$INPUT"; exit 0
P3
Headless CI — four required flags + dedup
--bare makes runs deterministic. stream-json makes them observable. Dedup prevents comment spam across PR runs.
+

Four flags that make CI production-grade: --bare (no runner state bleed), --output-format stream-json (every call loggable), --max-turns N (hard loop cap), --allowedTools "..." (technical whitelist).

Cost reality: Sonnet ~$0.02-0.05 per medium PR. 10 PRs/day team: ~$30-50/month total. Reserve Opus for scheduled weekly deep audits — not every-push reviews.

Dedup is essential at team scale. Without it, Claude comments the same security issue on every re-push. Fetch prior Claude comments, pass them in context, and explicitly tell Claude to only report new or still-unresolved issues.
.github/workflows/claude-review.yml
name: Claude Review
on:
  pull_request:
    paths: ['src/auth/**','src/payments/**']
jobs:
  review:
    runs-on: ubuntu-latest
    permissions: {contents: read, pull-requests: write}
    steps:
      - uses: actions/checkout@v4
        with: {fetch-depth: 0}
      - name: Dedup-aware Claude review
        env:
          ANTHROPIC_API_KEY: ${{secrets.ANTHROPIC_API_KEY}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}
        run: |
          PRIOR=$(gh pr view ${{github.event.pull_request.number}} \
            --json comments \
            --jq '.comments[]|select(.author.login=="github-actions[bot]")|.body' \
            2>/dev/null || echo "")

          claude -p "Review PR for security issues.
            PRIOR REPORTED (do not repeat): $PRIOR
            Output JSON: {issues:[{file,line,severity,
            desc,fix}],pass:bool,recommendation:string}" \
            --bare \
            --output-format json \
            --max-turns 8 \
            --allowedTools "Read,Grep,Glob,Bash(git diff:*)" \
            > review.json

          gh pr comment ${{github.event.pull_request.number}} \
            --body "$(jq -r '.result' review.json)"
P4
Prompt injection defense in agentic pipelines
External content can hijack agents. Sanitize before it enters context. Scope reads to what's necessary.
+

When Claude Code reads PR comments, issue bodies, or external file content, that content can contain instructions that redirect the agent. This is documented in OWASP's 2026 Top 10 for Agentic Applications as goal hijacking.

Safest defense: scope what Claude can read to exactly what the task requires. Feed Claude the git diff, not PR comments. If you need external content, sanitize it first — extract only structured data, not raw user-controlled text.
.claude/hooks/sanitize-input.sh (UserPromptSubmit)
#!/bin/bash
# Flag injection patterns before Claude sees the prompt
INPUT=$(cat)
PROMPT=$(echo "$INPUT" | jq -r '.prompt // empty')

PATTERNS=("SYSTEM:" "you are now" "ignore previous"
  "disregard" "new instructions" "override"
  "forget everything" "as an AI without")

for p in "${PATTERNS[@]}"; do
  if echo "$PROMPT" | grep -qi "$p"; then
    echo "{\"decision\":\"block\",
      \"reason\":\"Potential injection: '${p}'. Verify source.\"}"
    exit 2
  fi
done
exit 0
Scoped reads in CI
# Only source files — not PR comments or issue bodies
DIFF=$(git diff origin/main...HEAD | head -400)
claude -p "Review this diff only: $DIFF" \
  --bare \
  --allowedTools "Read(src/**),Grep,Glob(src/**)" \
  --max-turns 5
P5
Verification-aware prompting — rubric before code
Explicit done criteria as a checklist Claude must verify. Claude stops when all boxes are checked, not when it feels done.
+

Without done criteria: Claude decides what "done" means and declares done when it feels complete. With explicit criteria: Claude runs tests, checks types, validates specific behaviors — and doesn't stop until all boxes are checked.

The Stop hook (run tests before Claude can stop) is the automated version. The verification prompt is the manual version — use it when you need more nuanced criteria than "all tests pass."
Verification-aware task prompt
Implement the rate-limiting middleware.

DONE means ALL of the following are true:
[ ] pnpm test src/middleware/rateLimit.test.ts
    exits 0 with no failures
[ ] pnpm build exits 0, no TypeScript errors
[ ] Rate limit headers on every response:
    X-RateLimit-Limit, X-RateLimit-Remaining,
    X-RateLimit-Reset
[ ] Redis connection failure → graceful degradation
    to in-memory fallback (test: kill Redis mid-request)
[ ] Per-route configurable limits (not just global)
[ ] No secrets or connection strings hardcoded

Do not declare done until you confirm each box
by running the relevant verification command.

P1

The Agent SDK is the same runtime that powers Claude Code — exposed as a library your application controls. The query() function returns an async iterator; each message is a step in the agent loop.

Model routing is the primary cost lever. Opus for deep reasoning (orchestration, complex debugging). Sonnet for standard work (implementation, review). agent-runner.ts
import { query, ClaudeAgentOptions }
  from "@anthropic-ai/claude-agent-sdk";

function selectModel(task: string): string {
  if (/architect|plan|audit|complex/i.test(task))
    return "claude-opus-4-6";   // Deep reasoning
  if (/classify|extract|route/i.test(task))
    return "claude-haiku-4-5";  // High-volume narrow
  return "claude-sonnet-4-6";   // Safe default
}

async function budgetedRun(
  prompt: string, maxUsd = 0.50
): Promise {
  let spent = 0; let result = "";

  for await (const msg of query({
    prompt,
    options: new ClaudeAgentOptions({
      model: selectModel(prompt),
      systemPrompt: STATIC_SYSTEM_PROMPT, // Always cached
      allowedTools: ["Read","Grep","Glob",
        "mcp__internal__get_project_status"],
      maxTurns: 20,
    })
  })) {
    const m = msg as any;
    if (m.usage) {
      spent += (m.usage.input_tokens * 0.000003)
             + (m.usage.output_tokens * 0.000015);
      if (spent > maxUsd)
        throw new Error(`Budget exceeded: $${spent.toFixed(4)}`);
    }
    // Always check subtype, NEVER is_error (known bug)
    if (m.type === "result") {
      if (m.subtype === "error_max_turns")
        console.warn("Hit turn limit");
      result = m.result ?? "";
    }
  }
  return result;
}
P2
Intent-grouped MCP server — not a 1:1 API mirror
Design tools around tasks, not endpoints. One rich tool beats four primitives that the model has to orchestrate.
+

The primary leverage point at Level 4 is tool design, not model selection or prompt quality. Vague tool docstrings kill selection accuracy. Past ~15 tools, accuracy degrades measurably.

For APIs with hundreds of endpoints (AWS, Kubernetes, Cloudflare), use the dispatcher pattern: one tool accepts a script, runs it in a sandbox against your API, returns only the result. Cloudflare covers ~2,500 endpoints with two tools.

P3
Prompt caching — static content first, always
40-90% token cost reduction. The invariant: dynamic content in the system prompt breaks the cache on every call.
+

Prompt caching reuses previously computed KV attention tensors for repeated prompt prefixes. For agents that resend the same system prompt across dozens of API calls, caching eliminates the bulk of computation cost.

The invariant: the moment you put dynamic content (task IDs, timestamps, user names) into the system prompt, you break the cache on every single call. Cache-friendly means: system prompt never changes, tool definitions never change, current task is always at the end in the user message.

Minimum cache threshold: 1,024 tokens for most models. Design your system prompt to exceed this — usually easy given architecture context, tool descriptions, and examples.

Cache-friendly vs cache-breaking
// ✅ Cache-friendly: static prefix, dynamic suffix
const STATIC_SYSTEM = `
You are a security auditor for [Company].
[Role definition — never changes]
[Architecture context — never changes]
[Tool descriptions — never changes]
[Few-shot examples — never changes]
`; // Entire block cached after first call

for await (const msg of query({
  // Dynamic content goes in the user message:
  prompt: `Audit this file: ${filePath}\n${fileContent}`,
  options: new ClaudeAgentOptions({
    systemPrompt: STATIC_SYSTEM, // ✅ Cached
  })
}));

// ❌ Cache-breaking — miss on every call
const BAD_SYSTEM = `
You are auditing task: ${taskId}    // ❌
Requested by: ${user.name}          // ❌
At: ${new Date().toISOString()}     // ❌
`;
P4
Durable execution with phase checkpoints
Long-running tasks that survive interruption. Resume from the last successful phase, not from the beginning.
+

Standard agent runs fail silently on timeout, network error, or context overflow. Durable execution means the agent can be interrupted at any point and resumed without re-doing completed work.

The critical property: each phase must be idempotent. Running phase 3 twice produces the same result as running it once. This lets you safely retry failed phases without fear of duplicate records, double sends, or inconsistent state.
durable-runner.ts
import fs from "fs";
const CKPT = ".agent-checkpoint.json";

async function durableRun(phases: AgentPhase[]) {
  let ckpt = fs.existsSync(CKPT)
    ? JSON.parse(fs.readFileSync(CKPT,"utf8"))
    : { completedPhases: [], sessionId: undefined };

  for (const phase of phases) {
    if (ckpt.completedPhases.includes(phase.id)) {
      console.log(`Skip completed: ${phase.id}`);
      continue;
    }
    try {
      const result = await runPhase(phase, ckpt.sessionId);
      if (!ckpt.sessionId && result.sessionId)
        ckpt.sessionId = result.sessionId; // Persist for resumption

      ckpt.completedPhases.push(phase.id);
      fs.writeFileSync(CKPT, JSON.stringify(ckpt,null,2));
    } catch(e) {
      console.error(`Phase ${phase.id} failed. Re-run same command to resume.`);
      throw e;
    }
  }
  fs.unlinkSync(CKPT); // Clean up on success
}
P5
Governance — audit trail, security posture, shadow MCP
Every tool call logged. Centralized .mcp.json. Security built around hard boundaries, not prompt instructions.
+

Critical security truth: CLAUDE.md constraints are advisory only. --allowedTools is the only hard enforcement boundary. Design your security posture around technical limits, not prompt instructions that the model might not follow.

The Shadow MCP Problem: MCP adoption begins with a few developers and scales to dozens managing hundreds of connections — each configured locally with different credentials. No central visibility. Fix: commit a single .mcp.json to the repo. That file is the source of truth. Review changes to it like code changes.

Three non-negotiable principles:
1. Least privilege per task — each invocation gets only the tools it needs.
2. Human checkpoints at irreversible boundaries — production deploys, external sends, DB mutations.
3. Observability before autonomy — full audit trail before expanding agent permissions.

E1
Agent hooks — AI-driven verification decisions
Spawn a full subagent with tool access to make verification calls. More thorough than prompt hooks, slower.
+

Four hook handler types: command (shell script), HTTP (webhook POST), prompt (single LLM yes/no), and agent (spawns a subagent with Read/Grep/Glob to make verification decisions). Agent hooks can explore the codebase before deciding — something no other hook type can do.

Agent hook on Stop — verifies tests before allowing stop
{
  "hooks": {
    "Stop": [{"hooks":[{
      "type": "agent",
      "prompt": "Verify all unit tests pass. Run the test suite. Check for TypeScript errors. Return ok:true only if everything passes. Include failing test names in your reason if not.",
      "timeout": 120
    }]}],
    "SubagentStop": [{"matcher":"code-reviewer","hooks":[{
      "type": "agent",
      "prompt": "Review the code-reviewer's findings. Are they specific, actionable, and free of false positives? If the review is vague or flags correct code, return ok:false with reason.",
      "timeout": 60
    }]}]
  }
}

Use prompt hooks when input data alone is enough to decide (30s default timeout). Use agent hooks when the decision requires codebase exploration (60s default). Use command hooks for deterministic checks — they handle 80% of cases and are fastest.

E3
SubagentStop — cross-agent quality gates
Fire when any subagent completes. Block it from returning until quality passes. Keep it working.
+

SubagentStop fires when a subagent finishes, before it returns results to the main agent. Returning decision: "block" keeps the subagent running and delivers the reason as its next instruction. This enables automated quality loops without the main agent needing to orchestrate the retry explicitly.

SubagentStop hooks receive: agent_id, agent_type, agent_transcript_path, and critically last_assistant_message — the subagent's final output text, available without parsing the transcript file.

SubagentStop quality gate for code-reviewer agent
#!/bin/bash
INPUT=$(cat)
AGENT_TYPE=$(echo "$INPUT" | jq -r '.agent_type')
LAST_MSG=$(echo "$INPUT" | jq -r '.last_assistant_message')

if [ "$AGENT_TYPE" != "code-reviewer" ]; then exit 0; fi

# Require file:line references in every finding
if ! echo "$LAST_MSG" | grep -qE '[a-zA-Z]+\.[a-zA-Z]+:[0-9]+'; then
  echo "{\"decision\":\"block\",
    \"reason\":\"Review lacks file:line references. Every issue must cite exact location.\"}"
  exit 2
fi

# Require CRITICAL/IMPORTANT/SUGGESTION severity levels
if ! echo "$LAST_MSG" | grep -qE 'CRITICAL|IMPORTANT|SUGGESTION'; then
  echo "{\"decision\":\"block\",
    \"reason\":\"Review must use severity levels: CRITICAL/IMPORTANT/SUGGESTION.\"}"
  exit 2
fi
exit 0
E4
Async hooks — fire and forget without blocking
async: true runs hooks in background. Logging and backup never add latency to Claude's execution.
+

Released January 2026: async: true on any hook handler runs it in the background without blocking Claude's execution. Hook fires, Claude continues, hook result is ignored (success or failure doesn't affect the session).

Use async for: audit logging, session backups (PreCompact), Slack notifications, metrics collection. Never async for security gates — those must block synchronously. An async security hook provides zero protection.

Async logging alongside sync formatting
{
  "hooks": {
    "PostToolUse": [{"matcher":"Write|Edit","hooks":[
      {"type":"command","async":true,

Both hooks fire on the same event. The audit log writes asynchronously — zero latency impact. Prettier runs synchronously — Claude waits for clean output before continuing.

Compound engineering — the mindset that separates teams
Every session makes the next better. The six-month CLAUDE.md is worth more than any prompt technique.
+

Boris Cherny's philosophy: treat CLAUDE.md as a compound interest vehicle. Every wrong assumption Claude makes is an opportunity to prevent it permanently. Every repeated correction is a missed hook or skill.

The compound loop
WHEN Claude makes a wrong assumption:
  → Don't just fix the output
  → Ask: what to add to CLAUDE.md to prevent this?
  → Encode it immediately, commit with the fix

WHEN you correct the same issue twice:
  → It's not a CLAUDE.md problem — it's a hook problem
  → Make it deterministic, not advisory

WHEN a workflow recurs more than twice:
  → It's not a conversation problem — it's a skill problem
  → Package it once, invoke it forever

WHEN a session goes wrong:
  → Write the root cause to MEMORY.md
  → Add the WHY to CLAUDE.md so the next session
    starts with that knowledge already encoded

Result at 6 months:
  A CLAUDE.md + hooks + skills setup encoding
  6 months of your team's hard-won decisions.
  Every session starts smarter than the last.
  Teams running this loop compound.
  Teams that don't stay flat.

- Pattern matching over type casting — always
- Collection expressions: [..list] over new List(list)
- Nullable reference types enabled — no #nullable disable
- ConfigureAwait(false) on all library awaits
- Cancellation tokens on all async public methods

## Known wrong defaults
- NEVER use int for IDs — use Guid or strongly-typed IDs
- NEVER return Domain entities from API — use DTOs/response types
- NEVER catch Exception — catch specific exceptions
- NEVER use async void — use async Task
- ALWAYS use ILogger, never Console.WriteLine
- ALWAYS use ProblemDetails for error responses (RFC 9110)

## Testing conventions
- Test project mirrors src structure: Application.Tests, etc.
- Use xUnit + FluentAssertions + NSubstitute (not Moq)
- Integration tests use WebApplicationFactory + Testcontainers
- Test names: MethodName_StateUnderTest_ExpectedBehavior

CLAUDE.md — Blazor Web App (.NET 8+)
# [Project Name] — Blazor

## Identity
Blazor Web App .NET 8+. Render mode: [Server / WASM / Auto / Static SSR].
Architecture: component library in Shared/, pages in Pages/,
services registered in Program.cs via DI.
Fluent UI Blazor Components for UI primitives.

## Render mode — CRITICAL
This project uses: [Auto render mode]
- Static SSR: no interactivity, use for SEO-critical pages
- Server: real-time, SignalR connection, OnInitializedAsync runs on server
- WebAssembly: runs in browser, no direct server access
- Auto: Server first load, WASM on subsequent loads

## Commands
- Build: `dotnet build`
- Run: `dotnet run --project src/BlazorApp`
- Test: `dotnet test`
- Watch: `dotnet watch run --project src/BlazorApp`
- Publish WASM: `dotnet publish -c Release`

## Component conventions
- Page components: @page directive, in Pages/ folder
- Reusable components: in Components/ or Shared/
- Parameters: [Parameter] attribute, PascalCase names
- Cascading: [CascadingParameter] — use sparingly
- EventCallbacks: EventCallback not Action
- Dispose: implement IDisposable for event subscriptions

## State management rules
- Component state: private fields + StateHasChanged()
- StateHasChanged(): call only after async operations
- NEVER call StateHasChanged() in synchronous loops
- Shared state: scoped services or Flux/Fluxor pattern
- Server mode: scoped = per-circuit, NOT per-request

## Known wrong defaults
- NEVER use JSRuntime in OnInitializedAsync on Server mode
  (use OnAfterRenderAsync(bool firstRender) instead)
- NEVER inject HttpClient directly in Server mode
  (use IHttpClientFactory with named clients)
- ALWAYS use @key directive in foreach loops with components
- NEVER mutate [Parameter] values — use callbacks
- ALWAYS await InvokeAsync(StateHasChanged) from background threads
- Use CancellationToken in all async component methods

## JS Interop rules
- Use IJSRuntime.InvokeAsync for JS calls from C#
- Use DotNetObjectReference for callbacks from JS to C#
- Dispose DotNetObjectReference — it's a memory leak if not
- Prefer CSS/Blazor solutions over JS interop

Anti-patterns to add to CLAUDE.md
## Blazor wrong defaults — correct immediately

### Lifecycle
WRONG: using OnInitializedAsync for JS calls in Server mode
RIGHT: use OnAfterRenderAsync(bool firstRender) for JS interop
WHY: DOM doesn't exist during OnInitializedAsync on Server

WRONG: calling StateHasChanged() from a Timer callback directly
RIGHT: await InvokeAsync(StateHasChanged) from background threads
WHY: Blazor components must be updated on the render thread

WRONG: not disposing event subscriptions
RIGHT: implement IDisposable, unsubscribe in Dispose()
WHY: Server mode keeps components alive for the circuit lifetime

### Render mode confusion
WRONG: accessing HttpContext in WASM or Auto mode
RIGHT: HTTP calls via IHttpClientFactory, never HttpContext
WHY: HttpContext only exists on the server

WRONG: using scoped services as if they're per-request in Server
RIGHT: scoped services are per-circuit (entire user session)
WHY: Server mode uses SignalR circuits, not HTTP requests

### Component patterns
WRONG: @bind-Value without @bind-Value:event="oninput"
       when real-time validation is needed
RIGHT: use both when immediate feedback matters
WRONG: mutating [Parameter] properties directly
RIGHT: use EventCallback to notify parent of changes
